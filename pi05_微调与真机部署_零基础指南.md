> **面向**:第一次接触 VLA / 机器人学习的同学
> **目标**:在自采数据上微调 π0.5,部署到实验室真机完成任务
> **你的仓库**:https://github.com/Isaiah-WU/pi0.5-Fine-Tuning
> **v2 依据的三个已知条件**:训练用 A100/H100(全量微调)· 主从臂遥操作(ALOHA 系平台)· 数据自己采集
>
> **v2 核心变化**:确认走 ALOHA 路线后,数据转换、训练配置、真机部署三个阶段都有官方现成模板可以照抄——比通用路线省一半工作量。

## 0. 任务需求拆解:7 个子任务

- **① 环境搭建**:GPU 服务器装 openpi(uv + JAX);机器人端电脑装 ALOHA 软件栈(ROS)
- **② 跑通基线**:官方 checkpoint 推理 + simple_client 验证"服务器–客户端"链路
- **③ 数据采集**:主从臂遥操作录制 50–100 条 `episode_*.hdf5`
- **④ 数据转换**:用官方脚本 `examples/aloha_real/convert_aloha_data_to_lerobot.py` 一键转 LeRobot 格式
- **⑤ 训练配置**:在 `config.py` 里**照抄 `pi05_aloha_pen_uncap` 模板**,改成自己的数据集和 prompt
- **⑥ 微调训练**:从 `pi05_base` 全量微调(官方模板:batch 64 / 20000 步)
- **⑦ 真机部署**:官方 `examples/aloha_real/` 提供 Docker 一键启动;评测成功率

**必须先向学长确认的 4 件事(决定后面怎么走):**

1. 机械臂具体型号:标准 ALOHA(Trossen ViperX/WidowX)?还是国产同类(ARX、松灵等)?→ 决定归一化统计能不能直接复用官方的
2. 单臂还是双臂?→ 标准 ALOHA 是双臂 14 维状态(每臂 6 关节 + 1 夹爪);单臂需要在配置里补维度
3. 相机几个、装在哪?→ ALOHA 标配最多 4 个:`cam_high`(顶视)、`cam_low`(低位)、`cam_left_wrist`、`cam_right_wrist`(双腕),常见配置是 3 个(顶 + 双腕)
4. 录制软件存出来的是不是 ALOHA 标准 hdf5(`/observations/qpos`、`/action`、`/observations/images/cam_*`)?→ 是的话转换零成本;不是就写个小脚本对齐字段

## 4. 阶段二:主从臂采集数据(第 1–2 周)

**录制机制回顾**(你已经懂原理了):人掰主臂 → 从臂跟随;系统每帧记录 **从臂关节角 = state(qpos)**、**主臂关节角 = action**、所有相机画面。

**ALOHA 标准数据格式**(转换脚本期望的,采集时就按这个存):

```
episode_0.hdf5
├── /observations/qpos          # (T, 14) 双臂各 6 关节 + 1 夹爪
├── /observations/images/cam_high        # (T, 480, 640, 3)
├── /observations/images/cam_left_wrist  # 同上
├── /observations/images/cam_right_wrist # 同上(有 cam_low 也一并存)
└── /action                     # (T, 14) 主臂关节角
```

- 采样频率:**50 Hz**(ALOHA 标准,转换脚本默认 fps=50)
- 图像:480×640 RGB(可存压缩格式,脚本会自动解压)

**采集操作清单:**

- 和学长定死任务与那句英文 prompt(例:*"pick up the cup and place it into the box"*),以后训练、部署都用这一句
- 正式采集前空练 20–30 遍,操作流畅了再开录
- 数量:**50–100 条起步**;每条从自然的初始位置开始,物体位置/朝向刻意变化(还记得泛化吗——练习册题型越丰富,考试越稳)
- 刻意录 10–20% "失败后恢复"片段(抓偏了 → 重新对准抓起)
- 剔除:彻底失败的、中途长时间停顿的(模型会学会"发呆")
- 如果实验室录制软件产出的字段名和上面不一样:不用慌,写 20 行脚本把字段改名对齐即可

## 5. 阶段三:一键转 LeRobot 格式(第 2 周,半天搞定)

官方现成脚本,不用自己写:

```bash
uv run examples/aloha_real/convert_aloha_data_to_lerobot.py \
  --raw-dir /path/to/你的hdf5目录 \
  --repo-id isaiah/pi05_cup_task \
  --task "pick up the cup and place it into the box" \
  --mode video
```

- `--task` 就是写进每条数据的语言指令(= 你的 prompt)
- 转换完用 LeRobot 可视化工具抽查 3–5 条:视频能播、qpos/action 曲线连续合理

## 6. 阶段四:训练配置 = 照抄官方模板(第 2 周)

**好消息:不用从零写。** `src/openpi/training/config.py` 里已有 **`pi05_aloha_pen_uncap`** —— 官方"从 pi05_base 微调一个 ALOHA 真机任务"的完整示范。复制这个 TrainConfig,改四处:

1. `name`:改成 `pi05_my_task`(随意)
2. `repo_id`:改成你上一步转换出的数据集 ID(`isaiah/pi05_cup_task`)
3. `default_prompt`:改成你的任务指令原句
4. `exp_name` 相关按需调整;`weight_loader` 保持指向 `gs://openpi-assets/checkpoints/pi05_base/params` 不动

**归一化统计(norm stats)的 ALOHA 特殊规则:**

- 官方模板做法:**直接复用 `pi05_base` checkpoint 自带的 ALOHA 归一化统计**(`asset_id="trossen"`,assets_dir 指向 base checkpoint)—— 因为 π0.5 预训练时见过大量 Trossen ALOHA 数据,沿用它的统计兼容性最好
- 如果你们是标准 ALOHA(Trossen 臂):照抄模板即可,**连 `compute_norm_stats.py` 都不用跑**
- 如果是国产非标臂(ARX 等,关节范围不同):跑 `uv run scripts/compute_norm_stats.py --config-name pi05_my_task` 自算,细节见仓库 `docs/norm_stats.md`
- `LeRobotAlohaDataConfig` 里的 `adapt_to_pi` 参数(把关节/夹爪数值换算到 π 内部约定空间)**保持默认,不要动**

## 7. 阶段五:启动全量微调(第 3 周)

```bash
# 先用 1-2 条数据做过拟合测试(排查配置错误最快的方法):
# loss 几百步内快速下降 → 配置没问题 → 再上全量数据

XLA_PYTHON_CLIENT_MEM_FRACTION=0.9 uv run scripts/train.py pi05_my_task \
  --exp-name=exp001 --overwrite
```

- 官方 ALOHA 模板超参:**batch 64、20000 步**,先不要自己发明超参
- A100/H100 → 全量微调;显存不够再退 LoRA
- `tmux` 挂后台,wandb 盯 loss;checkpoint 存到 `checkpoints/pi05_my_task/exp001/<步数>/`

## 8. 阶段六:真机部署(第 3–4 周)

**部署前一次性配置**:编辑 `examples/aloha_real/realsense_publisher.py`,把相机序列号改成你们实际相机的(用 RealSense 工具查)。

**方式 A:Docker 一键启动(官方推荐,机器人端执行):**

```bash
export SERVER_ARGS="--env ALOHA --default_prompt='pick up the cup and place it into the box'"
docker compose -f examples/aloha_real/compose.yml up --build
```

**方式 B:手动三个终端(理解链路用):**

- 终端 1(机器人端):`roslaunch aloha ros_nodes.launch`(起机械臂和相机的 ROS 节点)
- 终端 2(GPU 服务器):起你自己的 checkpoint —

```bash
uv run scripts/serve_policy.py policy:checkpoint \
  --policy.config=pi05_my_task \
  --policy.dir=checkpoints/pi05_my_task/exp001/19999
```

- 终端 3(机器人端):跑 `examples/aloha_real/` 的主循环脚本,填服务器 IP

**若是非标准 ALOHA(国产臂、自定义控制接口)**:参考 `docs/remote_inference.md`,用轻量包 `openpi-client`(`packages/openpi-client/`)自写十几行控制循环:发 {图像, qpos, prompt} → 收 action chunk → 执行前 N 步 → 重新推理。

**安全规程(第一次上真机必读):**

- 低速模式 + 关节限幅,手放急停
- 先"干跑":只打印模型输出的动作不执行,确认数值在合理范围
- 桌面清空无关物品,人在机械臂工作半径外

## 9. 阶段七:评测与迭代(第 4 周起)

- **评测协议**:固定 20 次试验;**分两组报告**——分布内(物体摆在采数覆盖范围)和 OOD(故意摆到没采过的位置/换相似物体),学长一眼看出真实水平
- 失败模式 → 对策:
  - 整体成功率低 → 数据翻倍
  - 特定位置总失败 → 针对性补采该位置演示
  - 动作抖动 → 检查 50Hz 是否稳定、action chunk 执行策略(执行多少步再重推理)
  - 换措辞不听话 → 正常,用训练原句
- 每次实验记录:数据版本 / config / 训练步数 / 两组成功率 → 仓库实验日志

## 11. 常见坑清单(ALOHA 特供版)

- 克隆忘加 `--recurse-submodules`;安装忘加 `GIT_LFS_SKIP_SMUDGE=1`
- **相机序列号没改 / 相机接错位置 → 画面串了,模型乱动**(cam_high 接成 wrist 这种)
- 训练用了自算 norm stats、部署却加载了 base 的(或反过来)→ 动作幅度诡异
- `adapt_to_pi` 被改动 → 夹爪/关节数值空间对不上
- 采数 50Hz 不稳(掉帧、卡顿)→ 数据时间轴错乱
- 遥操作数据大量停顿帧 → 模型学会"发呆"
- 部署时 prompt 没用训练原句 → 单任务微调后对措辞敏感
- 机器人端连不上服务器 → 同网段?8000 端口防火墙?
- 显存 OOM → 确认 `XLA_PYTHON_CLIENT_MEM_FRACTION=0.9`,减 batch,最后才换 LoRA
- 部署时光照/曝光和采数差异大 → 相机设固定曝光/白平衡
- 官方示例正常、自己的配置崩 → 回到"1 条数据过拟合测试"逐项排查
