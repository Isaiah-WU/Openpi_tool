# π0.5 微调与真机部署 · 零基础完全指南

> **面向**:第一次接触 VLA / 机器人学习的同学
> **目标**:把 Physical Intelligence 的 π0.5 模型在自己采集的数据上微调,并部署到实验室真机上完成任务
> **你的仓库**:https://github.com/Isaiah-WU/pi0.5-Fine-Tuning(目前只有 LICENSE,第 10 节讲怎么组织它)
> **根据你的情况定制**:训练用 A100/H100(走全量微调路线)、演示数据需要自己遥操作采集

---

## 0. 任务需求拆解:学长的一句话 = 下面 7 个子任务

- **① 环境搭建**:在带 GPU 的服务器上装好 openpi(JAX 训练框架 + 依赖)
- **② 跑通基线**:下载官方 π0.5 checkpoint,先让推理跑起来,确认环境没问题
- **③ 数据采集**:用遥操作在真机上演示目标任务,录制若干条轨迹(相机图像 + 机器人状态 + 动作 + 语言指令)
- **④ 数据转换**:把采集的原始数据转成 **LeRobot 数据集格式**(openpi 的标准输入格式)
- **⑤ 训练配置**:为你的机器人写三样东西 —— 输入/输出变换(policy)、数据配置(DataConfig)、训练配置(TrainConfig),再计算归一化统计量(norm stats)
- **⑥ 微调训练**:从 `pi05_base` 出发全量微调,用 wandb 监控 loss
- **⑦ 真机部署**:GPU 服务器上起 policy server,机器人端跑 client,通过 websocket 收发观测/动作,真机评测成功率

**建议第一时间和学长确认的验收标准:**

- 具体任务是什么?(例如"抓起桌上的杯子放进盒子"——这句话以后就是模型的 prompt)
- 成功率要求多少?评测多少次?
- 需不需要做对比实验(LoRA vs 全量、不同数据量)?
- 机器人具体型号、遥操作方案、相机配置、控制接口(ROS?厂商 SDK?控制频率多少 Hz?)

## 1. 背景知识学习路线(建议 3–5 天,不要超过一周)

> 原则:**不要等全部看懂再动手**。先建立直觉,边跑代码边回头补理论。

### 1.1 第一层:建立直觉(半天)

- **什么是 VLA(Vision-Language-Action)模型**:输入 = 相机图像 + 机器人当前状态 + 一句自然语言指令;输出 = 未来一小段动作序列(action chunk,例如一次输出 50 步)。本质是"看图、听话、干活"的大模型。
- **什么是模仿学习 / 行为克隆**:人遥操作机器人做示范 → 模型学"看到这样的画面就做这样的动作"。你要做的微调就是行为克隆。
- **什么是微调(fine-tuning)**:π0.5 已在上万小时、多种机器人的数据上预训练过,"会"大量通用操作;你只需用几十~几百条自己任务的演示,让它适配**你的机器人 + 你的任务**。
- 📖 先读 Physical Intelligence 官方博客(图文并茂,最适合入门):
  - π0 介绍:https://www.physicalintelligence.company/blog/pi0
  - π0.5 介绍:https://www.physicalintelligence.company/blog/pi05

### 1.2 第二层:核心论文(1–2 天,只抓主干)

- **π0 论文**(https://arxiv.org/abs/2410.24164):重点看模型结构部分。理解三个词就够:**PaliGemma VLM 骨干**、**action expert + flow matching 输出连续动作**、**action chunk**。
- **π0.5 论文**(https://arxiv.org/abs/2504.16054):重点看它比 π0 强在哪 —— 开放世界泛化、分层推理(先想"下一步该做什么"再输出动作)。
- 选读:**Knowledge Insulation**(https://arxiv.org/abs/2505.23705)—— π0.5 用到的训练技巧,知道这个名词即可。
- ⚠️ flow matching 的数学不用深究,理解为"类似扩散模型的连续动作生成方法"就够用了。openpi 仓库里 π0.5 的训练和推理目前只支持 flow matching 头。

### 1.3 第三层:工具链(1–2 天,边用边学)

- **Linux**:ssh 登录服务器、`tmux`(训练要跑很多小时,必须会挂后台)、`nvidia-smi` 看显存
- **uv**:openpi 用 uv 管理 Python 依赖(替代 conda),会 `uv sync`、`uv run xxx.py` 即可(https://docs.astral.sh/uv/)
- **Git**:clone / branch / commit / push,了解 submodule 概念
- **LeRobot 数据格式**:HuggingFace 的机器人数据集标准(视频 + parquet 表格),浏览一遍 https://github.com/huggingface/lerobot 的 README
- **JAX**:❗不用专门学。openpi 已把训练封装好,你只改配置文件;报错了再针对性查。(2025 年 9 月起官方也提供了 PyTorch 实现,但新手建议直接走默认的 JAX 主线)
- **wandb**:注册个账号,训练时看 loss 曲线用

## 3. 阶段一:装环境 + 跑通官方示例(第 1 周)

```bash
# 1. 克隆(注意 --recurse-submodules,漏了会缺依赖)
git clone --recurse-submodules https://github.com/Physical-Intelligence/openpi.git
cd openpi

# 2. 安装 uv
curl -LsSf https://astral.sh/uv/install.sh | sh

# 3. 安装依赖(GIT_LFS_SKIP_SMUDGE=1 必须加)
GIT_LFS_SKIP_SMUDGE=1 uv sync
GIT_LFS_SKIP_SMUDGE=1 uv pip install -e .
```

**验证环境三步走:**

- **单次推理**:按 README 里的 Python 片段加载 `pi05_droid` 或 `pi05_base` checkpoint,喂一个样例观测,能输出 action chunk 即成功(checkpoint 会自动从 `gs://openpi-assets/checkpoints/...` 下载)
- **服务器–客户端链路**:不需要机器人 ——
  - 终端 1 起服务:`uv run scripts/serve_policy.py --env DROID`(监听 8000 端口)
  - 终端 2 模拟机器人:按 `examples/simple_client/README.md` 跑 simple client,发随机观测、收到动作,说明整条部署链路是通的
- **(可选)LIBERO 仿真评测**:跑一遍 `pi05_libero`,直观感受"模型控制机器人"的完整闭环
- 如果环境装不顺,官方提供 Docker 方案:见仓库 `docs/docker.md`

📦 **本阶段产出**:环境可用的记录/截图,写进你仓库的 README。

## 4. 阶段二:遥操作采集数据(第 1–2 周)

- **先和学长定死任务**:一句话能说清(这句话就是以后的 prompt),例如 *"pick up the cup and place it into the box"*
- **搭好遥操作链路后先空练**:操作越熟练,数据越干净
- **每一帧必须记录的内容**:
  - 所有相机的图像(RGB;分辨率不低于 224×224)
  - 机器人本体状态 state(关节角 + 夹爪开合度)
  - 动作 action(目标关节角 / 末端位姿 —— **必须和将来部署时发给机器人的指令是同一种定义**)
  - 该条轨迹对应的语言指令文本
- **数量**:单任务从 **50–100 条轨迹**起步;不够再加采(第 9 节的迭代环节)
- **多样性**:物体位置、朝向、光照、背景尽量变化;刻意包含一些"失败后重试恢复"的片段,策略会更鲁棒
- **质量要点**:
  - 控制/记录频率固定(常见 10–50 Hz,**把数值记下来**,配置和部署都要用)
  - 起始状态自然多样,不要每次都摆得一模一样
  - 剔除彻底失败、乱动、长时间停顿的轨迹(停顿帧太多,模型会学会"发呆")
- 原始数据先存成你方便的格式(hdf5 / npz / rosbag 都行),下一阶段再统一转换

## 5. 阶段三:转成 LeRobot 数据集格式(第 2 周)

- **模板**:`examples/libero/convert_libero_data_to_lerobot.py`,复制一份改成自己的
- 核心逻辑就一个循环:遍历你的原始数据 → 每帧 `dataset.add_frame({图像键, state, action, task 指令})` → 每条轨迹结束 `save_episode`
- **三个关键决策(记录在案,训练与部署必须一致!)**:
  - 相机键名(如 `base_rgb` / `wrist_rgb`)及数量
  - state 向量的维度和每一维的物理含义
  - action 用绝对关节角还是增量(delta)?夹爪那一维如何归一化(如 0~1)?
- 转换完成后,用 LeRobot 自带的可视化工具**抽查几条**:视频能播、状态/动作曲线合理、指令文本正确

## 6. 阶段四:写训练配置(第 2–3 周)

照抄 LIBERO 的"三件套",换成自己的机器人:

- **① 输入/输出变换**:在 `src/openpi/policies/` 下新建 `my_robot_policy.py`,仿照 `libero_policy.py` 写 `Inputs` / `Outputs` 变换类 —— 把你数据集里的键名映射到模型期望的 images / state / actions;缺少的相机视角按官方做法补零并 mask
- **② 数据配置 + 训练配置**:在 `src/openpi/training/config.py` 里仿照 `LeRobotLiberoDataConfig` 写自己的 DataConfig,再注册一个 TrainConfig(比如叫 `pi05_myrobot`),`weight_loader` 指向 `pi05_base` 的预训练权重
- **③ 归一化统计量**:

```bash
uv run scripts/compute_norm_stats.py --config-name pi05_myrobot
```

- ⚠️ 已知坑(官方 `docs/norm_stats.md` 有说明):某些几乎不动的维度 std ≈ 0,归一化后数值爆炸导致 loss NaN —— 可以手动修正 norm stats 文件

## 7. 阶段五:启动全量微调(第 3 周)

```bash
XLA_PYTHON_CLIENT_MEM_FRACTION=0.9 uv run scripts/train.py pi05_myrobot \
  --exp-name=exp001_first_run --overwrite
```

- 你有 A100/H100 → 直接**全量微调**;万一显存不够,退回 LoRA 配置(22.5GB 即可)
- **强烈建议的调试顺序**:先只用 1–2 条轨迹跑"过拟合测试"(几百步内 loss 应快速下降)→ 证明配置没问题 → 再上全量数据正式训练。这是排查配置错误最快的方法
- 用 tmux 挂后台;wandb 上盯 loss 曲线稳定下降即正常
- 训练步数参考官方示例配置(数万步量级);checkpoint 自动存到 `checkpoints/pi05_myrobot/exp001_first_run/<步数>/`

## 8. 阶段六:真机部署(第 3–4 周)

**架构**:GPU 服务器(policy server)←— websocket —→ 机器人电脑(client)

**服务器端**(在训练服务器或任意 >8GB 显存的机器上):

```bash
uv run scripts/serve_policy.py policy:checkpoint \
  --policy.config=pi05_myrobot \
  --policy.dir=checkpoints/pi05_myrobot/exp001_first_run/29999
```

**机器人端**(详读仓库 `docs/remote_inference.md`):

- 只需安装轻量客户端包 `openpi-client`(在仓库 `packages/openpi-client/` 下),**不需要**装训练环境
- 控制循环的核心就这十几行:

```python
from openpi_client import websocket_client_policy

policy = websocket_client_policy.WebsocketClientPolicy(host="服务器IP", port=8000)

while 任务未完成:
    obs = {
        # 相机图像键名、state —— 必须和训练数据完全一致!
        "…images…": 当前相机画面,
        "…state…": 当前关节角和夹爪,
        "prompt": "pick up the cup and place it into the box",  # 用训练时的原句
    }
    action_chunk = policy.infer(obs)["actions"]   # 一次返回一段动作(如 50 步)
    for a in action_chunk[:执行步数]:              # 常见做法:只执行前一部分就重新推理
        机器人执行(a)
```

- **平台对照**:ALOHA → `examples/aloha_real/`;Franka → `examples/droid/`;UR5 → `examples/ur5/`;其他机械臂 → 参考 `examples/simple_client/` 自己写(核心就是上面那段循环)
- **安全第一**:首次上真机 —— 低速模式、关节限幅、手放急停按钮;可先"只打印动作不执行"空跑一轮,确认数值范围合理

## 9. 阶段七:评测与迭代(第 4 周起)

- **定评测协议**:固定次数(如 20 次)试验,物体位置按预设网格或随机摆放,记录成功率与失败模式
- **常见迭代方向**:
  - 整体成功率低 → 加采数据(数量翻倍)
  - 某个特定位置/姿态总失败 → 针对性补采该情形的演示
  - 动作抖动/卡顿 → 检查控制频率与 action chunk 的执行策略(执行多少步再重新推理)
  - 换个说法就不听指令 → 正常现象,单任务微调后对措辞敏感,部署时用训练原句
- **每次实验记录四件事**:数据版本、config、训练步数、成功率 → 写进仓库实验日志

## 11. 常见坑清单(踩过的人都懂)

- 克隆忘加 `--recurse-submodules` → 缺依赖报错
- 安装忘加 `GIT_LFS_SKIP_SMUDGE=1` → 卡死或狂下大文件
- **训练与部署的观测键名 / 图像分辨率 / state / action 定义不一致 → 真机上模型乱动(最高频的坑!)**
- norm stats 某维 std ≈ 0 → 归一化爆炸、loss NaN → 手动修正
- 显存 OOM → 确认 `XLA_PYTHON_CLIENT_MEM_FRACTION=0.9`;减小 batch;最后手段换 LoRA
- 部署时光照/相机曝光与采数时差异大 → 成功率骤降;相机设固定曝光/白平衡
- 遥操作数据大量停顿帧 → 策略学会"发呆";采数动作保持连贯
- 机器人端连不上服务器 → 检查同网段、8000 端口防火墙
- 跑官方示例正常、自己的配置就崩 → 回到"1 条数据过拟合测试"逐项排查
